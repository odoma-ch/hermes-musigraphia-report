# Culture Knowledge Graph Exploration Report

## Overview
Large-scale semantic infrastructure developed within NFDI4Culture to interconnect heterogeneous cultural heritage and humanities research data — including musicology, art history, performance, archives, and audiovisual collections. It aggregates metadata from distributed repositories into a shared graph based on ontologies such as NFDIcore and the NFDI4Culture Ontology (CTO), enabling cross-collection discovery, SPARQL querying, interoperability, and AI-driven exploration of cultural data across institutions and domains.
Endpoint: https://nfdi4culture.de/sparql

### Key Observations
- **Dominant types** are from the **NFDI ontology** (`NFDI_0000003`, `NFDI_0000004`, `NFDI_0000005`) and **Schema.org** (`ImageObject`, `DataFeedItem`, `VisualArtwork`, `MusicEvent`, `MusicComposition`)
- **Culture-specific ontology** (`nfdi4culture.de/ontology/CTO_*`) is heavily used
- **Music domain** is prominent: `mo:MusicalManifestation` (1.59M), `schema:MusicEvent`, `schema:MusicComposition`
- **CIDOC-CRM** appears for cultural heritage objects (`E34_Inscription`, `E22_Human-Made_Object`)
- **FaBiO** (scholarly publishing) types appear for research outputs
- **VoID datasets** (10,532) suggest this is a federated/aggregated knowledge graph

- **Virtrdf** internal properties appear at lower counts — system-level metadata from the Virtuoso triple store

### Key Relationship Pattern Findings

1. **External Identity Hub**: `NFDI_0001006` is the dominant bridge property — both Organisations and Persons use it to link to millions of external authority records (Wikidata, ROR, RISM, GND, etc.), making it the primary cross-KG alignment mechanism.

2. **Organisation internal structure**: Orgs have rich nested relationships — `NFDI_0000213` (sub-organisation), `NFDI_0000182` (related organisation), `BFO_0000178` (organisational unit), `org:classification` (type), `schema:member` → role nodes (with `roleName`, `startDate`, `endDate`), and `schema:location` → `schema:geo`.

3. **Person–Project–Output chain**: Persons connect to projects via `NFDI_0001028` (project membership), which in turn links to contributors, keywords, datasets (`NFDI_0000116`), publications, and actions. `NFDI_0000117` (affiliation) and `NFDI_0000113` (research output) are the other key person-level hubs.

4. **Role reification**: Both `schema:member` (from Org) and `schema:memberOf` (from Person) point to intermediate role blank nodes carrying `roleName`, `startDate`, `endDate`, `sameAs` — a reified membership pattern.

5. **Topic/Research area links**: `schema:knowsAbout` and `NFDI_0000211` connect persons to research topics, which carry `owl:sameAs`, `NFDI_0001006`, `schema:name` — linking to controlled vocabularies.

6. **Serialisation pattern**: `rdfs:seeAlso` consistently points to `.json`, `.nt`, `.rdf`, `.ttl` serialisation endpoints for every entity — a systematic content-negotiation pattern throughout the KG.

### Key Findings from Step 5

1. **Virtuoso system data dominates the unfiltered scan** — the default graph includes ~130 Virtuoso internal `QuadMapFormat` entities before any domain data appears. This is a known artefact of Virtuoso's RDF storage layer.

2. **Multilingual labels**: Organisations carry both German and English `rdfs:label` values for the same entity — internationalisation is built in.

3. **Structured label strings**: Both events (`CTO_0001005`) and incipits (`CTO_0001024`) encode rich structured metadata directly in their `rdfs:label` literals (venue, date, time, key, tempo, catalogue numbers), making labels human-readable summaries of the entity.

4. **ARK persistent identifiers**: DataFeedItem URIs use the ARK (Archival Resource Key) scheme (`ark:/60538/`) — a library/archive standard for persistent identifiers, consistent with the cultural heritage domain.

5. **URI slug encoding**: Event URIs at `performance.musiconn.de` encode the full event description (series, venue, date, time) as a URL slug, providing human-readable, dereferenceable identifiers.

## 10. Key Structural Patterns

### 10.1 Serialisation Pattern
Every named entity carries four `rdfs:seeAlso` links to content-negotiated serialisations:
- `https://nfdi4culture.de/resource/<ID>.json`
- `https://nfdi4culture.de/resource/<ID>.nt`
- `https://nfdi4culture.de/resource/<ID>.rdf`
- `https://nfdi4culture.de/resource/<ID>.ttl`

### 10.2 External Identity Hub
`NFDI_0001006` (11.1M triples) is the primary cross-KG alignment property, linking entities to ROR, Wikidata, RISM, ZDB, GND, and other authority systems. It functions as the KG's `owl:sameAs` equivalent for external identifiers.

### 10.3 Reified Membership Pattern
Both `schema:member` (from Organisation) and `schema:memberOf` (from Person) point to intermediate blank-node role objects carrying:
- `schema:roleName` — the role title
- `schema:sameAs` — authority link for the role
- `schema:startDate` / `schema:endDate` — tenure dates
- `schema:member` / `schema:memberOf` — the linked person or organisation

### 10.4 Data Feed Aggregation
`schema:DataFeedItem` (2.4M instances) wraps external resource URIs via `schema:item`, with `schema:dateCreated` and `schema:dateModified` timestamps. This is the KG's mechanism for aggregating records from contributing data sources.

### 10.5 Music Incipit Encoding
`CTO_0001024` incipits encode musical notation using the **Plaine & Easie Code** standard (via `CTO_0001063`), alongside structured metadata for clef (`CTO_0001061`), time signature (`CTO_0001060`), key signature (`CTO_0001062`), and tonal centre (`CTO_0001064`). This enables music search by melodic content.

### 10.6 Blank Node Usage
A significant proportion of entities (ImageObjects, MusicalManifestations, VisualArtworks, Places, Locations) exist only as blank nodes — typed but without URIs. These serve as structured value nodes linked from named entities, rather than independently dereferenceable resources.

### 10.7 Virtuoso System Data in Default Graph
The Virtuoso triple store's internal `virtrdf#QuadMapFormat` entities (~130 instances) are present in the default graph alongside domain data. These are system-level format descriptors and should be filtered out in domain queries using `FILTER(!STRSTARTS(STR(?s), "http://www.openlinksw.com/"))`.
